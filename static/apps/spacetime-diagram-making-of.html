The making of the Spacetime Diagram applet

A lightly edited transcript of the conversation that built the spacetime-diagram applet.

A curated transcript of the conversation (with Claude Code) that produced the 1+1-dimensional special-relativity spacetime-diagram applet for tao-web, from first sketch to publication.

How to read this. The point is to convey the process, not to reproduce the raw log, so:

  • Terence's messages are shown, mostly verbatim. A few of the earliest ones, and a handful of short clarifications, are condensed from a session summary (the full text had already scrolled out of the log) — these are marked (condensed).
  • Claude's replies — mainly code edits to two files, spacetime.js (physics core) and spacetime-diagram.html (the app) — are replaced by short italic summaries of what changed. The actual diffs are omitted.
  • Unrelated threads from the same sessions (other website edits, reboots, meta-questions) are left out.

The whole thing was built incrementally over several sittings, kept uncommitted for local testing until the very end. Physics was verified with small headless Node unit tests throughout.

1. The idea, and a plan

Terence (condensed):

I'm thinking of coding a new applet for the website, after some local testing: a spacetime-diagram simulator for 1+1-dimensional special relativity. The interface should feel a bit like Euclid: The Game — you construct events (points in spacetime), light rays (null lines), inertial objects (timelike lines), lines of simultaneity (spacelike lines), and observers (timelike polygonal paths). Eventually I'd like a sophisticated click-and-drag editor, and the ability to "experience" an observer's worldline; but as a first stage, entering objects by coordinates in the currently active frame — with an API to switch to other frames — is fine. The goal is to visualize the standard relativity "paradoxes" (twin paradox, pole and barn, …) as an aid for undergraduates, in the same spirit as the linear-algebra and complex-analysis apps. Let's workshop the idea first and come up with a staged implementation plan.

Claude proposed reusing the site's existing canvas machinery (plane.js / applet.css), sketched the Minkowski conventions (natural units c = 1, so light is at 45°), and laid out a staged plan: (1) a coordinate-entry MVP with switchable frames, (2) a drag editor, (3) a construction language, (4) subjective observer experience, (5) dynamics and appearance. It then asked two questions: how should the frames be laid out, and what to build first after the MVP.

Terence (condensed — answers to those questions):

I like the dual-panel idea, except that both panels can be "active": by default both start off as the same frame, but each can be manipulated independently. At some stage one can run an animation tracing an observer's worldline — one frame stays a fixed inertial frame with the observer's subjective state a little coloured dot moving along the worldline, while in the other frame the dot stays pinned at the origin and the rest of the universe moves around it (cf. the heliocentric and geocentric views of the solar system). For the next step after the MVP: the drag interface.

Claude built Stage 0 + a first Stage 1 MVP: two independently boostable panels, coordinate entry for all object types, per-panel frame controls, and a twin-paradox preset.

2. "The travellers are tachyons" — and a pivot

Terence:

At a later stage, we can give observers and inertial objects (rest-frame) colour and (inertial) mass, and be able to view their (red-shifted) colour, relativistic mass, energy, and momentum somehow (e.g. by mouseover), but this can be a Stage 5.

I think your time and space conventions are inverted — the travellers are tachyons right now!

Eventually we need the ability to rename objects, etc. I'm thinking an Inkscape-type interface where one might have several modes selectable from a sidebar, such as clicking on an observer to select it and then dragging its nodes one at a time. Actually we're basically coding a dual-panel Inkscape editor now that I think about it.

Claude fixed the inverted axes — the drawing code had been mapping space and time the wrong way, so timelike worldlines rendered as if faster-than-light — and reframed the roadmap around the "dual-panel Inkscape editor" idea, moving colour/mass/energy to Stage 5.

3. Building the editor (Stage 1)

Terence:

Great! I have a clearer idea of what I want for the GUI now, so I'd like to refine Stage 1.

The dual-frame elements are already great, except: at any time one of the two frames should have the "focus" (left by default; clicking a frame focuses it), indicated subtly — e.g. a slightly thicker border. The focus frame is the one whose coordinates are displayed.

The "Scene" option would, in the Inkscape model, fit under a menu bar (like "File / Open…"), freeing screen real estate. On a left sidebar, Inkscape-style icons: a pointer to select any object in either frame, plus one icon per object type. With the pointer selected, clicking an existing object brings up editable text fields above the frames — the object's name and its parameters as seen in the focused frame (so clicking the other frame instantly changes the numbers). The observer editor will need one box per (t, x) vertex, with buttons to insert/remove points, plus a Delete button. With an object mode selected, you create by clicking (events), click-and-drag (worldlines / light rays / simultaneity lines, like circles in the Möbius app), or multi-clicking (observers, like a polygonal path in Inkscape). A snap-to should map a click near an existing object onto it. Physically illegal objects (faster-than-light, etc.) simply aren't created.

If a rich enough editor exists, Stage 3 isn't needed.

Claude built the full editor: menu bar (Scene presets, object picker), the tool sidebar (pointer + five object types), the focus-frame concept, frame-aware editable properties (observer as per-vertex rows with insert/remove), rename, delete, and click/drag/multi-click creation with snapping. Stage 3 was dropped.

Terence:

A small feature request: a "swap" button (maybe a double-arrow icon) between the two frames that interchanges them (and swaps the focus).

Added.

Terence:

Observer names should default to "Observer 1", "Observer 2", … rather than "undefined". Also, the edges of an observer path aren't drawn until it's finalized — this object type needs an explicit "Complete" button (or Enter), because the GUI can't otherwise know when you're done. Draw the unfinished path in grey; on completion it turns colourful and is added.

The legend at the bottom should distinguish objects by shape/style, not colour (dots = events; dashed = light rays and simultaneity lines; solid = timelike), since objects will soon have colours.

Two more sidebar tools: a "hand grab" mode to translate a frame by dragging (ideally with a clenching-hand cursor), and a "boost" mode to Lorentz-boost a frame by dragging within a cone.

Claude added the grey-then-commit observer flow with a Complete button, a shape-based legend, and the hand (pan) and boost tools. A follow-up clarified the boost math: even when the drag's start and end points are at different Minkowski distances from the origin, apply the boost that maps the ray through one to the ray through the other. Another aligned the Delete key with the Delete button.

Terence:

The edit pane has variable height (it grows as an observer gets more vertices), which pushes the frame canvases down and makes multi-click placement annoying — give it a fixed height with a scrollbar. The boost sliders are indexed to the (hidden) lab frame rather than to each other; per the principle-of-relativity discussion, each slider should read relative to the opposite frame. The two align buttons both point right — replace them with an up/down arrow pair around the swap button. While laying an observer path, show a grey rubber-band edge to the cursor. Add zoom in/out buttons (also scroll-wheel and +/− keys on the focused frame). And display each frame's spacetime-origin offset in the other frame's coordinates.

Claude reworked the layout (fixed-height scrolling edit pane, later moved below the canvases), made the sliders and origin readouts relative, added the rubber-band observer edge, zoom controls, and a consolidated align/swap control cluster.

4. Relativity by design: hiding the lab frame

(Claude had made the frame readouts relative to each other and removed any way to select the "lab" frame directly; Terence confirmed the intent.)

Terence (condensed):

Yes exactly. The lab frame is needed for the back-end implementation, but should be invisible to the end user, who can only access the two visible frames. With no way to read off lab coordinates, they'd have no way to determine them — 100% in the spirit of the principle of relativity.

The lab frame stayed a pure backend construct: no "select lab" option, no lab label; each panel shows its velocity relative to the other (0 = co-moving), and the rest-frame dropdown only offers objects' rest frames.

Terence (condensed):

As a variant of snap-to, the interface can try to make the object being constructed physically possible. A light ray that isn't quite 45° snaps to the nearest 45° line; a line of simultaneity drifting toward lightlike/timelike has its slope top out just short of 45°.

Creation became self-correcting: light rays snap to the nearer null direction; timelike/spacelike lines are clamped toward, but never across, the light cone; illegal observer steps are refused.

5. Colour, redshift, mass, energy

Terence:

In the edit panel add a colour (a rainbow, except events and simultaneity lines can be any colour), and inertial mass (inertial and observer objects only, default 1). For inertial objects, light rays, and observers, frequency and wavelength are tied to colour and displayed (nm, THz); editing any one updates the others. This colour is the rest-frame colour (inertial objects and observers) or the current-frame colour (light rays) — store light-ray colour in the lab frame but render in the current frame. So these objects show different colours in different frames due to redshift, and an observer even shows different colours per segment. Also display the relativistic mass/energy and momentum (c = 1), and observed colour, as non-editable fields — for observers, one interval per consecutive pair of nodes.

Claude added the colour model (hex for events/simultaneity; a visible-spectrum wavelength — named dropdown + rainbow slider + nm + THz — for the rest), rest mass, and derived read-only E, p, γ and observed-colour fields, with the Doppler physics (gamma, dopplerNm, wavelengthToRgb, etc.) added to the core and unit-tested. Several clarifications shaped the Doppler model:

  • a named colour is pinned to one canonical frequency (even though "red" really spans a band);
  • colours that come out ultraviolet or infrared display as the nearest human-visible colour;
  • the shift is from the viewpoint of an observer on the frame's x = 0 line, so a worldline flips from blue (approaching x = 0) to red (receding) as it crosses — so each drawn segment is split at x = 0, and the frequency/colour readout splits into approaching/receding parts.

Terence:

When using the hand or boost tool, keep the previously focused object in focus, so I can watch its statistics change in the edit pane as I use the tool.

Done — the boost/pan tools now update a held selection's readouts live.

Terence:

The labels changing colour with the shift was more visually annoying than I expected. Make all objects labelled by their rest colour, except light rays, which should all use a single constant colour (say black). Changing the frame changes the segment colours but not the text.

Applied: names render in rest colour (light rays in black); only the worldline strokes Doppler-shift.

6. Direct manipulation (Stage 2)

Terence:

I think we're ready for Stage 2 — editing a selected object by dragging. Events: pick up and drag. Inertial / light / simultaneity lines: drag the reference point (translate, keeping orientation) or click the line body and drag (re-orient, like creation). Observers: move individual nodes, but an interior node must stay within the forward light cone of the previous node and the backward light cone of the next (both displayed); an endpoint is constrained by just one cone.

Claude added grab-handles and edit-drag modes for every object type, with the observer's neighbouring light cones drawn as constraints during a node drag, and live statistic/colour updates throughout.

7. Light rays as segments; light energy; "Show c"

Terence:

A significant change to the light-ray object. Right now it's an infinite null line with one marked point. For illustrating emission/absorption (and reflection = absorption of one ray + emission of another), I'd rather light rays be null line segments. Internally: an initial spacetime point, a left/right orientation, and the elapsed lab-frame time — hidden from the user, who sees the initial and final points and the orientation in the focus frame. On creation, the moving endpoint snaps to the light-cone boundary of the initial point (illegal if it's in the past). When editing, both endpoints are editable and stay on that null cone.

Two more, roughly independent. To illustrate energy/mass conservation, give light rays an energy and momentum (stored in the lab frame): either is editable, but E = pc forces them equal. (We don't enforce Planck's law — a ray can be many photons.) And a Settings ▸ Show c toggle (like a Word-Wrap tick) that appends powers of c to the relevant quantities: x, v, p get a "c"; energy gets "c²"; frequency, wavelength, and dimensionless quantities are unchanged.

Light rays became null segments ({initial point, direction, lab elapsed}), with cone-snapping creation/editing and an E = p energy carried in the lab frame and shown in the focus frame. The Settings menu gained "Show c", appending the right powers of c in the edit pane only.

Terence:

A bug: in light-ray mode, once a ray exists I can only start a new one if the first click isn't on a previous ray — because clicking a ray selects it. But clicking a ray that's already focused should start a new ray, so I can draw a chain of reflections.

Fixed: clicking the already-focused object in a creation mode starts a fresh object rather than re-selecting.

8. Proper time

Terence:

Give each inertial worldline and observer an assigned proper time at its initial point (default 0). In the observer edit panel, show each vertex's proper time next to its (t, x), as non-editable text. And in pointer mode, when an inertial object or observer is selected, hovering over it should reveal the proper time at that point as hovertext.

Added a tau0 per worldline, a read-only τ column in the observer editor, and proper-time hovertext at the nearest point on the selected worldline (all frame-invariant).

9. The hyperbola (a sixth object)

Terence:

Create a sixth object type: a hyperbola, all points at a constant spacetime interval from a centre — a timelike-cone hyperbola, a spacelike-cone hyperbola, or the light cone itself, depending on the sign of the interval. It's the Minkowski analogue of a circle, with a GUI like the circles in the Möbius app. Store the centre in lab coordinates and an interval (any sign). Edit-panel attributes: colour (arbitrary, not redshifted), centre (in the focus frame), and the (Lorentz- invariant) interval. Creation: click-drag A → B to draw all points at the same interval to A as B.

One small thing: the natural icon for the hyperbola tool is a hyperbola — but I'd used that for the boost tool. Give the boost tool a different icon, maybe boosted t- and x-axes.

Claude added the hyperbola object (constant-interval "Minkowski circle", two branches via cosh/sinh, light-cone degenerate case), Möbius-style drag creation, and centre/interval editing; the boost tool got a crossed-axes icon. A follow-up set the line style by causal character:

The hyperbola's style can match other objects of the same causality: negative interval → timelike → solid; positive → spacelike → dotted; zero → light cone → dashed.

Terence:

Minor GUI: the "rest frame: rest frame…" dropdown is redundant — drop the extra label; show "(no rest frames)" when none exist. Add a Settings option besides "Show c": "Apply redshift" (default on) — when off, every object shows its rest colour regardless of velocity. And in observer segments you're using β for apparent velocity; please use v.

All three done.

10. Subjective experience of time (Stage 4)

Terence:

Ready for Stage 4: subjective experience of time. A subjective-experience button (a YouTube-style "play" triangle) in the sidebar. Select an object, and a YouTube-like timeline appears in the edit panel with a current-time marker; you can play in real time or scrub, with a speed control (0.5×, 2×, …). Observers have a finite timeline; inertial objects a doubly-infinite one (so add from/to boxes, default the marked point's τ ± 5).

While an object is selected, the non-focused frame becomes the co-moving frame at the observer's current subjective position and velocity (defaulting, at a node, to the velocity before the jump). Pressing play animates it (with all Doppler shifts). At a velocity transition I'd like a smooth slew through intermediate velocities rather than a discontinuous jump.

Two features that can be deferred: (a) a panel recording what the observer "sees" in its backward light cone — each visible object's apparent displacement, velocity, frequency, wavelength, energy, momentum (objects appearing/disappearing as the cone sweeps); and (b) marks on the timeline where the worldline meets another object (an event, a simultaneity line, another observer), taking care with degeneracies (a colocation over an interval marks a band, not a point).

Claude built the playback engine: a scrubber with play/pause and speed, driven by proper time; the non-focused panel switches to the co-moving frame, with rapidity slewed smoothly through observer nodes; frames are saved on entry and restored on exit. Several clarifications refined it:

  • first clicking a worldline sets the subjective location to that click point (not the start);
  • the non-focused subjective frame gets a soft pink "subjective" border/label, and clicks on it are disabled;
  • if no object is focused, the subjective frame simply doesn't appear;
  • in the focused frame, a large hollow "You are here" ring marks the subjective event; clicking the worldline recenters there.

Terence:

When first clicking a worldline, set the subjective location to the click location immediately — right now it defaults to the initial location (fine for the dropdown selector, but the click should jump straight there rather than needing a double-click).

Fixed.

Terence:

Let's now implement the deferred features.

Claude added the two deferred pieces:

  • a backward-light-cone "seen" panel: past null rays from the subjective event to every other worldline give emission events, each reported with apparent position (± = direction), relative velocity, Doppler-shifted colour, E and p; a fast or bending source can show two images (left/right), and a colocated object shows a single "•" entry;
  • red timeline marks where the experienced worldline crosses another object — ticks for point crossings, bands for colinear overlaps.

Terence:

When in subjective mode, the name tag of the experienced object should ride with the hollow circle, not stay pinned to its start point. And add a timeline tick for when an event E is observed subjectively — i.e. when E's forward light cone meets the worldline (this generalizes the colocation tick; when E is actually on the worldline, only one tick).

Both added — the label follows the "you are here" ring, and events register both colocation and light-cone-observation ticks.

Terence:

The tooltips for the ticks are reporting an error message: "Value must be less than or equal to 6.98…".

That was the range input's native validation bubble leaking through the tick overlay. Claude moved the marks to a strip above the slider and clamped the slider's value to the timeline bounds.

11. Import / export

Terence:

I'd like to import/export the applet state as JSON, via the Scene menu. The state is the objects (all their data) plus the two frames (the current tool needn't be stored) — the existing Scenes are essentially already this. Export downloads a JSON; import opens a file picker and converts it back, returning an error if it fails, with physicality checked during import. JSON is pure text and the parser won't execute anything, so the security risk is acceptable.

Claude added Scene ▸ Export/Import JSON: export serializes the lab-frame objects and both frames to a downloadable file; import fully validates structure and physicality (|β| < 1, future-timelike observer steps, positive light-ray durations) before touching the scene, alerting on any failure.
